## 1. Краткий итог

В рамках проверки были протестированы два варианта системы:

1. Основной проект `diploma` - микросервисная архитектура с API Gateway и асинхронной обработкой событий через Kafka.
2. Отдельный проект `diploma-sync` - микросервисная архитектура с тем же общим смыслом, но с синхронными HTTP-вызовами между сервисами.

Основной вывод:

Kafka-версия устойчивее к отказам зависимых сервисов. При отключении `notification-service` пользовательский сценарий создания заказа продолжает работать: заказ создается, а событие остается в Kafka и может быть обработано после восстановления сервиса уведомлений. В синхронной версии тот же сценарий ломается: API Gateway возвращает ошибку `502`, потому что цепочка запроса напрямую зависит от доступности `notification-service`.

Это наглядно показывает практическую пользу Kafka в микросервисной архитектуре: Kafka снижает связанность сервисов, позволяет переживать временные сбои отдельных компонентов и делает систему более устойчивой под нагрузкой и при частичных отказах.

## 12. Kafka-версия: пользовательский тест через API Gateway

Основной проект тестировался через API Gateway `http://localhost:8080`, то есть запрос шел как от реального пользователя.

Важно: Kafka-версия выполняет больше работы, чем упрощенный sync-стенд:

- регистрация или авторизация пользователя;
- JWT cookie;
- role-based access control;
- Redis rate limit;
- HMAC/internal-auth между сервисами;
- полноценная доменная модель заказа;
- запись статусов и истории;

## 13. Сравнение Kafka и sync

Главное сравнение:

| Сценарий | Sync через Gateway | Kafka через Gateway |
|---|---:|---:|
| Нормальная работа | Заказы создаются | Заказы создаются |
| Высокая нагрузка | Работает, но появляются локальные edge-ошибки | Работает, выполняя более тяжелый полный сценарий |
| `notification-service` отключен | Создание заказа падает с `502` | Создание заказа успешно, `201` |
| `user-service` отключен | Создание заказа падает с `502` | Обязательные синхронные зависимости все еще критичны |
| Поведение при временном сбое уведомлений | Пользователь получает ошибку | Пользовательский сценарий продолжается |
| Связанность сервисов | Высокая | Ниже за счет Kafka |
| Устойчивость к частичным отказам | Ниже | Выше |

Вывод:

Kafka не делает систему "магически неубиваемой". Если падает обязательная синхронная зависимость, например сервис пользователей, сценарий все равно может сломаться. Но Kafka сильно помогает там, где операция может быть разорвана на основную часть и побочные действия. В данном проекте создание заказа является основной операцией, а уведомление - побочной. Поэтому при падении сервиса уведомлений Kafka позволяет не ломать пользовательский сценарий.

## 14. Что именно показывает разница

В синхронной архитектуре цепочка выглядит так:

```text
Пользователь -> API Gateway -> Order Service -> Notification Service
```

Если `Notification Service` недоступен, весь запрос может завершиться ошибкой, даже если сам заказ технически можно было бы создать.

В Kafka-архитектуре цепочка выглядит так:

```text
Пользователь -> API Gateway -> Order Service -> Kafka -> Notification Service
```

`Order Service` создает заказ и публикует событие. `Notification Service` читает событие из Kafka отдельно. Если сервис уведомлений временно недоступен, создание заказа не блокируется.

Именно это было подтверждено нагрузочным тестом:

- sync-версия при отключенном notification-service: 1000 из 1000 пользовательских запросов завершились ошибкой `502`;
- Kafka-версия при отключенном notification-service: 1000 из 1000 созданий заказа завершились успешно со статусом `201`.

## 16. Ограничения тестирования

У тестирования есть несколько важных ограничений, которые стоит честно указать в дипломе:

1. Тестирование выполнялось локально, а не в Kubernetes-кластере.
2. Результаты latency зависят от локального компьютера, Docker Desktop, базы данных и состояния системы во время прогона.
3. Sync-стенд является отдельной реализацией для сравнения архитектурного подхода, а не полной копией production-версии основного проекта.
4. Kafka-версия выполняет больше бизнес-логики и проверок, поэтому ее normal latency нельзя напрямую сравнивать с упрощенной sync-версией как чистый benchmark.
5. Самое важное сравнение - отказоустойчивость при падении notification-service, потому что именно здесь Kafka проявляет архитектурное преимущество.

## 18. Готовый текст вывода для диплома

В ходе тестирования была проверена работа микросервисной системы в двух вариантах: с асинхронным взаимодействием через Kafka и с синхронным HTTP-взаимодействием между сервисами. Функциональное тестирование основного проекта подтвердило корректность ключевых пользовательских сценариев: регистрация и авторизация пользователей, работа ролей, создание и обработка заказов, предложения исполнителей, чаты, уведомления, жалобы и административные действия.

Нагрузочное тестирование показало, что обе архитектуры способны обрабатывать большое количество запросов при штатной работе сервисов. Однако наиболее важное различие проявилось при моделировании отказа `notification-service`. В синхронной реализации создание заказа зависело от прямого вызова сервиса уведомлений, поэтому при его отключении все пользовательские запросы на создание заказа завершались ошибкой `502`. В реализации с Kafka создание заказа продолжало успешно выполняться: система возвращала пользователю статус `201`, а событие о создании заказа передавалось через Kafka для последующей обработки.

Таким образом, использование Kafka позволило снизить связанность микросервисов и повысить отказоустойчивость системы. Основной пользовательский сценарий не блокируется из-за временной недоступности сервиса, отвечающего за побочные действия. Это подтверждает целесообразность применения брокера сообщений в проектируемой микросервисной архитектуре.

Критический сценарий, ради которого сравнивались Kafka и sync-подход, подтвержден: Kafka-архитектура сохраняет работоспособность пользовательского сценария создания заказа при падении сервиса уведомлений, тогда как синхронная архитектура возвращает ошибку.
